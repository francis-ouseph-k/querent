# Background
# ──────────
# The original implementation required bitsandbytes for NF4 4-bit
# quantisation (QLoRA) during training and evaluation.  This is the standard
# approach for fine-tuning large models on consumer GPUs and works correctly
# on Ampere (sm_80) and earlier NVIDIA architectures.
#
# The Problem
# ───────────
# On NVIDIA RTX 50-series "Blackwell" GPUs (compute capability sm_120 —
# e.g. RTX 5060 Ti, RTX 5070, RTX 5080, RTX 5090):
#   • bitsandbytes NF4 dequantisation kernels have not been validated for
#     sm_120 and produce numerically corrupt outputs (all-NaN logits).
#   • fp16 mixed precision is less stable than bf16 on Blackwell Tensor Cores
#     which are natively optimised for bf16 and tf32.
#   • device_map="auto" triggers an unstable multi-device dispatch path in
#     transformers even on single-GPU Blackwell systems.
#
# The Workaround (active in trainer.py and evaluator.py)
# ───────────────────────────────────────────────────────
# Standard LoRA on a full bf16 base model is used instead of NF4 QLoRA.
# This eliminates the bitsandbytes dependency entirely.
# Validated across 4 training runs on RTX 5060 Ti (8 GB VRAM, sm_120,
# CUDA 13.1, driver 591.74) — stable loss curves, no NaN logits.
#
# Trade-offs
# ──────────
#   ┌─────────────────────┬───────────────────────────┬──────────────────────┐
#   │                     │ Legacy (NF4/bitsandbytes) │ Blackwell-safe (bf16)│
#   ├─────────────────────┼───────────────────────────┼──────────────────────┤
#   │ bitsandbytes needed │ Yes                       │ No                   │
#   │ VRAM (base model)   │ ~4–5 GB (4-bit)           │ ~6–7 GB (bf16)       │
#   │ Adapter size        │ ~50 MB                    │ ~100 MB              │
#   │ Numerical stability │ Lower (fp16 + NF4)        │ Higher (bf16 native) │
#   │ Blackwell (sm_120)  │ ✗ NaN logits             │ ✓ Validated          │
#   │ Ampere (sm_80)      │ ✓ Validated              │ ✓ Compatible         │
#   │ Turing (sm_75)      │ ✓ Validated              │ ✓ Compatible (8+ GB) │
#   └─────────────────────┴───────────────────────────┴──────────────────────┘
#
# Reverting to the legacy (NF4/bitsandbytes) implementation
# ──────────────────────────────────────────────────────────
# If you are on a non-Blackwell GPU and want to restore NF4 QLoRA:
#   1. Uncomment the bitsandbytes line in the [LEGACY] section below.
#   2. In trainer.py, comment out the [BLACKWELL-SAFE] model load block
#      and uncomment the [LEGACY] NF4 block (clearly marked in the file).
#   3. In trainer.py SFTConfig, restore fp16=True, bf16=False.
#   4. In evaluator.py, comment out the [BLACKWELL-SAFE] block and
#      uncomment the [LEGACY] NF4 block in _load_model_and_tokenizer().
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

trl>=0.8.6,<0.10.0                  # SFTTrainer — supervised fine-tuning wrapper

# ── bitsandbytes — NF4 quantisation ──────────────────────────────────────────
#
# [LEGACY] NF4 QLoRA — Ampere (sm_80) / Turing (sm_75) and older
# Uncomment to enable NF4 QLoRA on non-Blackwell GPUs.
# Reduces VRAM from ~6–7 GB (bf16) to ~4–5 GB (4-bit NF4).
# Produces ~50 MB adapters vs ~100 MB for bf16.
# DO NOT enable on RTX 50-series (Blackwell / sm_120) — causes NaN logits.
# Also revert trainer.py and evaluator.py model load blocks to [LEGACY].
#
# bitsandbytes>=0.43.0,<0.44.0      # [LEGACY] 4-bit NF4 quantisation
#                                   # Windows note: use --prefer-binary if standard fails
#                                   # pip install bitsandbytes --prefer-binary
#
# [BLACKWELL-SAFE] bitsandbytes NOT installed (active configuration)
# bf16 standard LoRA is used instead.  No quantisation library required.
# trainer.py and evaluator.py load the base model via torch_dtype=bfloat16.
